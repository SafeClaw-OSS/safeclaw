## S2 decisions (resident hudsucker proxy stack: components A, B, C, F-proxy-side)

- **hudsucker 0.24.1 wired** (default features = decoder/rcgen-ca/rustls-client, pure aws-lc-rs, NO OpenSSL). Deps added: `hudsucker = "0.24"`, `http-body-util = "0.1"` (for the bounded body scan). The process rustls crypto provider is installed ONCE in `main::run_daemon` (`aws_lc_rs::default_provider().install_default()`, error ignored). The proxy has its OWN upstream TLS client (`with_rustls_connector`, webpki roots) — unrelated to the daemon's reqwest clients (which still target real hosts directly; no HTTPS_PROXY leaks into the daemon).
- **Two listeners live.** `main::run_daemon` spawns `proxy::serve` (PROXY_PORT 23294, hudsucker MITM) detached and keeps axum (CONTROL_PORT 23295) in the foreground; a proxy exit logs and leaves the control plane up. Smoke-verified: both bind, CA generates once (`<state_dir>/ca.pem` + `ca.key` `0600`), `curl -x http://127.0.0.1:23294 http://sc.probe/` → `{"proxy":true,"safeclaw":<ver>}`, and a non-connection HTTPS host blind-tunnels with its REAL cert (selective MITM by the unlocked host union).
- **Approval-cache key gained host.** `rule_approvals` is now `(conn, rule, method, host)`; `evaluate_request_policy`/`record_ask_approval` take a `host: &str`. `PolicyContext` gained `host: Option<String>` (plain field, not serde) — the proxy stamps it at op-create; `approve.rs` reads `pc.host` (falls back to `scope.host`) for the cache write. E2E-5 oracle added to the `state.rs` scope test (approve host A → host B re-asks).
- **Raw-connection allow bootstrap (touches `approve.rs::bootstrap_cache_from_view`).** Raw connections (`service: None`) default to `allow` (global floor), and the proxy resolves an allow request straight from the session cache (it holds no grant to open the vault), so their secret bytes must be resident. Added a reverse-index of `native_secrets`: a raw conn `c` owns the bare key `c` (sole) and/or `c:<ROLE>` keys — matches `secret_address` — case-insensitive (env-safe lowercase conn vs possibly upper-case stored key). **S3 alignment:** `sc set K --host h` must store the raw secret at bare `<conn>` (sole) or `<conn>:<ROLE>`; a different address needs this reverse-index adjusted.
- **Widen ceremony (component C) is SESSION-LEVEL.** Host-anchor miss to a public host → DENY 403 + a `Custom("widen-host")` pending op (scope `{connection_id, host, etld1}`) whose approve arm (`approve.rs`) calls `state.widen_connection_host` = appends the FQDN to the in-memory routing snapshot so the retry passes the anchor. **Durable persistence into `aux.connections[<conn>].hosts` is NOT wired** — it needs a daemon-side aux write (parked with the per-item write cutover). Integrator/console must persist the widen (the console's connection-edit already writes aux). Anchor miss to a private/metadata host → hard 403, no widen.
- **OAuth `exposes` not minted.** The mint (`resolve_auth_value`) returns only the access token; a role-qualified oauth phantom (e.g. codex `__sc__x__account_id__`) returns a clear 4xx `exposes_unsupported`. Follow-up lives in the mint, not the proxy.
- **Multi-secret ASK connections:** only the primary secret is stashed post-approval (`resolve_use_primary` resolves `op.act.target`), so a non-primary role phantom on an *ask*-level multi-secret connection can't resolve (portal loops). Allow-level multi-secret works (all roles are in `allow_secrets`). Edge; single-owned Basic pairs (username+token) are allow-level in practice.
- **Custom (`aux.services`) oauth:** `resolve_auth_value` + `service_env_key` read the COMPILED registry only, so a *custom* oauth service's mint/portal-target won't resolve; custom DIRECT services work (hosts/policy resolve via `custom_services`). Noted for a follow-up if custom oauth becomes real.
- **Body scan bound:** buffered + scanned only when content-type is text-ish (json/text/form/xml) AND content-length ≤ 1 MiB; larger/binary/unknown-length bodies stream through unscanned (phantoms live in headers/URL/Basic).
- **Auth strip:** `rewrite_headers` always drops `Proxy-Authorization` + hop-by-hop; substitutes a phantom inside `Authorization` (Bearer raw, or Basic decode→sub→re-encode) and keeps it; strips an `Authorization` that received NO substitution (the agent's own creds can't shadow the injected one). `content-length` is dropped only when the body was rewritten (hyper re-derives it).
- **h2/gRPC gap:** the proxy advertises `http/1.1` to the MITM'd client (hudsucker `http2` feature NOT enabled). Known, documented, not blocking.
- **Vault binding:** `should_intercept` reads the vid from the CONNECT `Proxy-Authorization: Basic base64("<vid>:")` and stores it on the (per-connection-cloned) handler, inherited by inner-request clones. Resolution order: explicit vid → single unlocked vault → 4xx (`ambiguous_vault` / `no_vault`). Intercept decision = host ∈ `host_in_any_unlocked_union` (all unlocked vaults, not vid-scoped, so a wrong/missing vid still MITMs and returns a precise error).

- **Raw-connection storage contract (aligns with S2's reverse-index in `bootstrap_cache_from_view`):**
  - `sc set K --host h` → conn id = `lower(K)`; the secret is stored at the BARE key `K` (so `name_lc == conn_lc` matches the sole-secret reverse-index). `lower(K)` must be a valid phantom segment (`[a-z0-9_]`, no `__`); if not (e.g. `K` has a `-`), `sc set` errors and points at `sc connect` rather than silently transforming the id and breaking resolution.
  - `sc connect NAME --secret KEY` → each NEW secret stored at `<NAME>:<KEY>` (reverse-index `<conn>:<role>`). Sole secret ⇒ short phantom `__sc__NAME__`; several ⇒ `__sc__NAME__<key>__`.
  - `--use-existing KEY` claims an ALREADY-stored bare secret and is only valid when `lower(KEY) == NAME` (the single-secret promote of a `--no-broker` item) — the one case that fits the raw reverse-index without a copy. Documented in the flag help + errors clearly otherwise.
- **`sc connect` non-TTY secret provisioning (smallest resolution of the map's `--secret KEY` ambiguity):** `--secret KEY=VALUE` provides a new secret inline; a bare `--secret KEY` prompts a hidden value ONLY on a TTY (errors off-TTY telling the user to use `KEY=VALUE`); `--use-existing KEY` needs no value. Fully-interactive path loops (host prompts → secret KEY prompts → hidden values), mirroring the console custom form.

Shared creation helper `cli/conn.rs` (id/host validation + `aux` mutation) is used by both `sc set --host` and `sc connect`.
